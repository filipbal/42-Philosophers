Quick Pre-Evaluation Guide for the Dining Philosophers Problem

The Setup

Philosophers (threads) sit around a table (shared memory)

Forks (mutexes) between each philosopher

Each philosopher: THINK ‚Üí EAT ‚Üí SLEEP ‚Üí repeat

Death condition: If no food within time_to_die milliseconds

üö® Critical Failure Points (Instant 0)

  • Segmentation fault or any crash
  • Compilation errors or norm violations
  • Data races (inconsistent behavior across runs)
  • Memory leaks (check with valgrind)
  • Philosophers eating without both forks
  • Multiple death messages or simulation continuing after death
  • Global variables used

üîç What Evaluators Check

Output Format Verification

Correct format: timestamp_in_ms philosopher_id action
Example: 0 1 is thinking, 1 1 has taken a fork, 310 2 died

Thread Safety

  • No overlapping or garbled print statements
  • Consistent behavior across multiple runs
  • Proper mutex usage for forks and shared data
  • Death detection within 10ms of actual death

Your Code Analysis

  • Good: Proper deadlock prevention (even philosophers take right fork first)
  • Good: Accurate timing with ft_usleep and busy-waiting
  • Good: Clean separation of concerns across files
  • Good: Proper mutex protection for shared data
  • Good: Single philosopher edge case handled
